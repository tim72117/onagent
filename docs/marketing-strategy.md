# onagent 行銷策略草案

> **文件性質**：這是一份**策略草案**，不是已經拍板執行的計畫。onagent 目前是
> pre-launch / early access 階段的個人（或極小團隊）專案——真實的 production
> 環境已經跑在 Cloud Run（`onagent.shuttle.tools`），console、CLI（`onagent`）、
> Agent Bridge SDK 都是可以真的動手串接的東西，但計費 / 訂閱 / 用量配額
> 完全還沒做（見 `docs/subscription-usage-quota-design.md`）。這份文件據此
> 假設：**現階段目標是社群心智佔有率與早期採用者，不是營收**，所有建議都
> 以「一個人、業餘時間、沒有行銷團隊、沒有付費預算」為前提設計。文中出現的
> 時程、指標都是預估的起始假設，不是已驗證的承諾。

## 1. 產品現況：這份策略是根據什麼寫的

寫行銷策略前，先誠實列出現在手上實際有什麼、沒有什麼——避免整份文件變成
套在任何一個「AI SaaS」上都能用的空話。

**已經是真的、可以讓人動手試的部分：**

- Production 環境：Go 後端 + `apps/landing`、`apps/console` 兩個內嵌前端，
  跑在 Cloud Run，掛了自訂網域 `onagent.shuttle.tools`（見
  `docs/deployment.md`）。這不是 staging，是會計費的真實雲端資源。
- Landing page 已經有真實文案，標題是「幾分鐘，讓你的網站長出 AI 助手。」
  ——這句話本身已經是不錯的一句話定位，後面第 2 節會延伸討論。
- Console（`onagent.shuttle.tools/app`）有完整的開發者工作流程：登入
  （含 CLI device-code 授權頁 `CliAuthPage.tsx`）、建立 App、Schema
  Editor 定義 tool 參數、**Playground**（不用寫任何前端程式碼，直接在
  console 裡跟自己定義的 tool 對話測試，`apps/console/src/Playground.tsx`
  有清楚註解說明這是刻意設計成「不需要真實前端站台就能先感受一次」）。
- CLI（`onagent`）：`go install github.com/tim72117/onagent/backend/cmd/onagent@latest`
  可用（module path 問題已修好），且已經有一個**內建 Claude Code Skill**
  （`.claude/skills/onagent-cli-setup`）——對於本來就在用 Claude Code
  的開發者，這是零安裝門檻：不需要裝 Go 工具鏈、不需要等編譯，直接請
  Claude 用 skill 裡預先編好的執行檔操作。這是一個具體、稀缺、目前
  同類產品幾乎沒人做的差異化切入點（詳見第 4 節）。
- `@onagent/bridge`：瀏覽器端 SDK，`examples/react-demo/tools.yaml` 有一組
  現成、敘事完整的範例 tool 定義——`fill_search_form`（填搜尋框）、
  `navigate_to_page`（換頁）、`highlight_element`（在畫面上框出元素）、
  `click_checkout_button`（按下結帳按鈕完成訂單）。這四個 tool 串起來
  剛好就是一支「AI 幫使用者搜尋、跳轉、標示重點、最後按下結帳」的完整
  demo 影片腳本，不需要另外發想 demo 情境——**它已經寫在 repo 裡了**。
- 安全模型是可以攤開來講、經得起技術受眾檢視的：per-app API key + Origin
  header allowlist，且是 fail-closed（沒設定 allowed origin 的 app 會
  拒絕所有連線，不是預設放行）。`docs/security-and-transport.md` 甚至
  誠實比較了 Google Analytics 的 gtag.js 傳輸設計、指出哪裡能借鏡（stub
  queue 緩衝、`sendBeacon` fallback）、哪裡不適用（WebSocket handshake
  不受瀏覽器 CORS 保護，所以 Origin 檢查必須在伺服器端做，不能指望瀏覽器
  幫忙擋）。這種願意公開技術判斷過程的文件，本身就是對開發者受眾最有效的
  行銷素材之一（詳見第 5 節）。

**還沒做、講策略時要誠實避開的部分：**

- 沒有計費 / 訂閱 / 方案分級。`docs/subscription-usage-quota-design.md`
  開頭就寫明「onagent 目前完全沒有計費 / 訂閱 / 配額機制」，現在只是一份
  設計提案。**這代表現階段完全不該做付費轉換導向的行銷**（例如「限時
  優惠」「立即訂閱」），任何現在進來的使用者事實上都是在用一個沒有用量
  上限、沒有 SLA 承諾的免費環境——這件事要嘛不提，要嘛就大方承認「現在
  是 early access，用量限制之後才會加上，現在進來的人會被當作核心早期
  使用者對待」。
- 沒有 per-session 真正的授權 token（目前身份驗證只到 WebSocket
  handshake 那一次 API key 檢查），`docs/security-and-transport.md`
  自己也列了這是正式 production-facing 前要補的項目。技術文件可以誠實
  討論這件事（開發者受眾其實會因為誠實而加分），但行銷文案不該做出
  「企業級安全」這種現在還撐不住的承諾。
- 沒有自動化測試套件、沒有多平台編譯好的 CLI（目前只內建
  `onagent-windows-amd64.exe`，其他平台的開發者要嘛用 `go install`、要嘛自己
  `git clone` + `go build`）。這代表任何「三分鐘上手」的 demo 素材，最
  穩妥的做法是先用 Windows 或已經有 Go 工具鏈的環境錄，或是明確走
  Claude Code Skill 那條路（那條路才是真正接近零安裝）。

這份現況盤點決定了後面整份策略的基調：**不是「怎麼包裝一個還沒做完的東西」，
而是「手上已經有一個可以真的動手玩的產品，怎麼讓對的一小群人先找到它、
玩起來、講出去」**。

## 2. 定位與訊息

### 一句話定位

Landing page 現有標題「**幾分鐘，讓你的網站長出 AI 助手。**」本身已經
掌握了核心賣點（快、且是「長」在你自己的網站上，不是外掛一個東西），但
對外部技術受眾（Hacker News、Twitter 上的開發者）需要更精準地把「不是
聊天機器人」這件事講出來，因為這正是最容易被誤解、也最容易被競品混淆的
一點：

> **onagent 讓使用者用自然語言操作你網站原本就有的功能——不是加一個
> 聊天視窗，是讓 AI 呼叫你自己寫的 JS 函式，去點你的按鈕、填你的表單、
> 跳你的頁面。**

英文版（給 Hacker News / Twitter 用）：

> **Let users operate your app in plain English — onagent's browser SDK
> lets an LLM call your app's real functions, not a bolted-on chatbot.**

這句話的每個詞都是刻意選的，對應到要打的認知戰場：

- 「操作你網站**原本就有**的功能」「你**自己寫的** JS 函式」——正面
  對打「這是不是又一個聊天機器人外掛」的直覺誤解。這是最大的認知門檻，
  多數人看到「AI + 你的網站」第一反應是 Intercom/Drift 那種右下角聊天
  泡泡，onagent 要在三秒內把這個預設印象打掉。
- 「呼叫」「點」「填」「跳」——用動詞而非名詞堆疊功能表，讓沒讀過任何
  文件的人也能想像畫面：AI 真的在操作介面，不是在生成一段文字回答。

### ICP（目標客群）

不是「所有想加 AI 功能的公司」，範圍要收得比這個窄很多，早期階段窄準比
廣泛更重要：

- **一個人或小團隊在做 SaaS 產品**，介面本來就已經有明確、可被函式化的
  操作（表單、篩選、多步驟流程），想讓使用者用講的完成操作，但不想從零
  刻一套 LLM tool-calling 的基礎設施（handshake、工具 schema 生成、
  prompt-to-tool-call 的推論串接、前後端協議）。
- **技術背景是前端 / 全端**，看到「YAML 定義 tool → 自動生成 TypeScript
  handler 介面」會立刻懂價值在哪，不需要被教育什麼是 tool calling
  ——他們大概率已經在別的情境用過 OpenAI/Anthropic 的 function calling
  或聽過 MCP，onagent 的價值主張建立在「你已經懂這個概念，我幫你把
  「浏览器端怎麼接」這一半基礎工程都做完了」。
- **已經在用 Claude Code 或類似 AI 輔助開發工具的開發者**——這群人是
  Claude Code Skill 那個切入點的直接受眾，「不用裝 Go、不用等編譯，
  直接請 Claude 用 skill 裡的 CLI 幫你操作 onagent」對這群人是真正的
  零門檻，也是目前同類產品幾乎沒人做的差異化。

**現階段刻意不追的客群**：企業採購決策者、需要 SOC2/SLA 保證的大型客戶、
非技術背景的「AI 賦能」需求方。理由很直接——現在沒有配額、沒有正式的
per-session 授權，這些客群一旦認真評估就會問出現在答不出來的問題，浪費
早期有限的心力在無法轉換的對話上。

## 3. 市場脈絡：為什麼現在，以及要跟誰做出區隔

2026 年中的 LLM tool-calling 生態，跟 onagent 最初設計時相比已經明顯
擁擠——這不代表機會消失，但代表**定位訊息必須主動避開被誤認成別的東西**，
否則會被開發者受眾快速歸類進錯的抽屜、然後忽略。

需要明確區隔的兩類鄰近產品：

- **通用瀏覽器自動化工具**（browser-use、Steel、Playwright MCP 等）：
  這些工具讓 AI agent 操作一個**外部、獨立的瀏覽器實例**去瀏覽「任意」
  網站——本質上是把網頁畫面（或 accessibility tree）餵給 LLM，讓它像人
  一樣用滑鼠鍵盤操作一個它不認識、不屬於你的頁面。onagent 的模型完全
  相反：**沒有外部瀏覽器，沒有畫面辨識，AI 呼叫的是嵌在使用者自己頁面
  裡、開發者自己寫的真實 JS 函式**。onagent 不是「幫 AI 學會用瀏覽器」，
  是「讓開發者自己的網站聽得懂自然語言指令」。這個區別要在任何對外文案
  的前 10 秒內講清楚，否則會被歸類成「又一個 browser automation」，
  完全跑錯戰場。
- **MCP（Model Context Protocol）**：MCP 是現在最主流的「LLM 怎麼接外部
  工具」標準化協議，多數技術受眾已經聽過。onagent 現階段**不是**在跟
  MCP 打對台，而是解決 MCP 生態目前沒有標準答案的一塊：**瀏覽器端、
  即時、跟使用者當下畫面狀態綁定的 UI 操作**。MCP server 多半跑在
  後端、操作的是資料庫/API/檔案系統這類無狀態資源；onagent 解決的是
  「AI 需要知道使用者現在畫面長怎樣、然後即時操作使用者眼前這個 DOM」
  這種強狀態、強即時性的場景。文案上與其正面比較（容易被讀成「我們
  比 MCP 好」，招來不必要的辯論），不如聚焦講清楚「這是給瀏覽器端
  即時 UI 操作用的」，讓懂 MCP 的人自己判斷這是互補還是替代。

這個市場脈絡的實務意義：**onagent 的內容行銷不該跟 browser-use 這類
工具搶「AI 操作網頁」這個關鍵字**（搜尋量會被巨大的通用工具流量稀釋、
也會被誤解成同類），而該搶「讓 AI 操作『我自己的』網站/UI，用『我自己
的』函式」這個更窄、但精準命中 ICP 的敘事角落。

## 4. 渠道策略

原則：**每個渠道都要對應一個具體的「第一步動作」，不是「去這個平台
發文」這種空泛指示。** 以下按投入產出比排序，不是按平台知名度排序。

### 4.1 Hacker News（Show HN）—— 最高槓桿，一次性事件

這是對的第一戰場：onagent 是「developer 拿來就能玩」的產品（有 console
Playground、有現成 demo tool YAML），完全符合 Show HN 的核心要求——
「連到一個真的能互動的東西，不是行銷頁」。

具體做法（根據 Show HN 的實際機制與過往成功案例，例如 Supabase 用
「an open source Firebase alternative」這個標題拿到超過 1,100 個
upvote，是史上最高分的 dev tool 發文之一）：

- **標題直接、具體、不像新聞稿**：例如「Show HN: Let an LLM click
  through your own web app's real UI, not a chatbot widget」或更貼近
  技術動作的版本「Show HN: onagent – give Claude/Gemini real
  function-calling access to your page's DOM」。避開「革命性」「顛覆」
  這類形容詞，HN 社群對這類詞非常敏感，容易被當成推銷文案直接忽略甚至
  被 flag。
- **連結指向可以馬上玩的東西**，不是 landing page。理想選項是連到一個
  已經串好 demo-app.yaml 那組 tool（搜尋 → 換頁 → 標示元素 → 結帳）的
  react-demo 公開部署，讓讀者不用建帳號就能先看到 AI 真的在操作畫面；
  次選是直接連 GitHub repo（`github.com/tim72117/onagent`），因為 HN
  讀者本來就傾向想先看程式碼再決定要不要試。
- **發文時機**：週二到週四美東上午 9 點到中午這段時間流量與活躍審核者
  最多；如果想避開最主戰場的競爭、鎖定更利基的早期受眾，週日晚間美東
  7 點左右也是有效的替代時段。
- **前 60 分鐘全程盯著留言**，每一則技術質疑都認真回——尤其會有人問
  「這跟 MCP/browser-use 差在哪」，直接用第 3 節整理好的區隔說法回答，
  不要迴避比較，迴避反而顯得心虛。
- **絕對不要**自己或請人 vote brigading（找人集中衝票）——HN 有偵測
  機制，一旦被抓到帳號會被封，且會連累未來所有發文永遠被降權。
- 只做一次，做之前先確保 demo 真的穩、Cloud Run 服務能撐住流量尖峰
  （前台可能單日湧入 5,000 到 30,000 不重複訪客），失敗一次要再等
  很久才有臉重新發。**建議把 Show HN 排在「前三個月」時間軸的中後段**
  （見第 6 節），而不是現在馬上做——先在小範圍渠道驗過一輪、修掉最明顯
  的 friction 之後再打這張一次性的牌。

### 4.2 Twitter / X —— 持續性渠道，demo 影片是主力彈藥

不是「持續發文互動」這種空話，而是具體素材類型：

- **主打格式：15-30 秒的螢幕錄影 GIF/影片，畫面是真實網頁，文字輸入
  一句自然語言指令，畫面上的表單自己被填、按鈕自己被按下**。這正是
  `demo-app.yaml` 已經寫好的敘事——录一次「幫我找戶外用品 → 跳轉到
  該分類頁 → 框出建議商品 → 完成結帳」的完整流程，這一支影片可以重複
  使用在 Twitter、Show HN 留言區、Product Hunt、landing page 上，
  是全渠道通用的核心資產，值得花時間錄到滿意為止。
- **技術決策串**：不是「onagent 上線了」這種公告文，而是拆解
  `docs/security-and-transport.md` 或 quota 設計文裡已經寫好的技術
  判斷過程，例如一串「WebSocket handshake 不受 CORS 保護，為什麼要
  在伺服器端做 Origin allowlist、而且是 fail-closed」的技術串——這類
  內容天然會被其他做類似基礎設施的開發者轉發，且不需要另外想內容，
  文件已經寫好了，只是換成社群媒體的敘事節奏。
- **Build in public 更新**：進度不是「又修了一個 bug」這種流水帳，而是
  「這禮拜想清楚了配額系統要怎麼設計，決定用 event log 而不是計數器，
  原因是……」這類有觀點、有取捨的更新——這正是 quota 設計文裡已經有的
  內容，換句話說 build-in-public 的素材成本極低，因為技術決策文件
  已經先寫好了，只是要摘要成推文長度。
- 務實預期：持續一年的規律發文，多數個人開發者帳號成長到約 500-5,000
  followers、直接歸因的付費轉換是個位數到低雙位數——這是業界對
  solo-founder build-in-public 的一般區間，不是悲觀預測，是設定合理
  期望值以免中途因為「怎麼還沒紅」而放棄。

### 4.3 GitHub —— repo 本身就是渠道

- Repo 目前是 `github.com/tim72117/onagent`，README 已經寫得很清楚
  （架構圖、quick start、status 章節誠實列出「還沒做什麼」）——這件事
  本身不用大改，但決定公開發布時機時，**README 的「Status / what's
  not built yet」章節要保留**，不要為了行銷好看而刪掉。開發者受眾對
  誠實列出限制的專案信任度明顯更高，這比包裝成「什麼都做完了」更有
  轉換力。
- 開一個 **GitHub Discussions**（不是 Issues——Issues 是給 bug/功能
  請求，Discussions 才適合「有人問『這個能不能整合 Vue』」這種開放式
  交流），這是目前完全零成本、且天然跟「已經在看 repo 的人」對話的
  管道，比另外建 Discord 更適合現階段的規模（見下方 4.4 的取捨判斷）。
- Star 數與 fork 數本身會被 Show HN/Twitter 流量帶動，不需要另外
  「衝 star」——但可以在 demo YAML、examples 目錄持續補充更多真實情境
  （例如一個電商 checkout、一個 dashboard 篩選器），讓 clone 下來的人
  有更多可以直接抄的範例，這比任何招募 star 的話術都有效。

### 4.4 Discord / Slack —— 現階段刻意不開，先觀察需求再決定

多數「該不該開 Discord」的建議是無條件說「開」，但對 onagent 現在的
規模（pre-launch、個人維護）要誠實評估成本：一個沒人維護、沒人回覆的
Discord 伺服器比完全沒有還傷品牌——會被新加入的人看到「上一則訊息是
三週前」，直接降低信任感。

務實建議：**先用 GitHub Discussions 撐住早期的問答需求，等出現以下
任一訊號再開 Discord**：

- 同一批問題在 Discussions/Twitter DM 反覆出現，代表有即時互動的需求；
- 有真實使用者主動問「有沒有社群可以加」；
- 每週有穩定超過個位數的新增 GitHub star/使用者，代表社群基數足以撐起
  一個不會冷清的即時聊天室。

這不是「Discord 沒用」，而是「開了沒人維護的 Discord」比「暫時沒有
Discord」更傷——資源有限時，把心力集中在一個能好好維護的渠道，好過
攤開三個都半殘的渠道。

### 4.5 dev.to / 部落格文章 —— 內容重複利用，不是額外負擔

不需要另外想部落格題目，把已經存在的技術文件改寫發布即可：

- `docs/security-and-transport.md` 改寫成一篇「Why WebSocket handshakes
  aren't protected by CORS, and what that means for embedding a
  bidirectional SDK on third-party sites」——這種標題本身就精準命中會
  對 onagent 感興趣的讀者（做類似跨站 SDK 的開發者），且完全不用重新
  做研究，文件已經在 repo 裡。
- `docs/subscription-usage-quota-design.md` 裡對 Stripe meter events、
  token bucket、event log vs. 計數器 race condition 的討論，本身就是
  一篇獨立成立的「usage-based billing 系統設計」技術文章，跟 onagent
  產品本身的推廣關聯度較低，但能建立作者本人在後端系統設計上的可信度
  ——這類「延伸技術信譽」文章的定位是軟性的，目的是讓讀者記住這個人
  /這個專案「講技術是講真的」，不是每篇都要直接導流到產品頁。

### 4.6 Product Hunt —— 排在 Show HN 之後，且要先有種子受眾

Product Hunt 的實際機制對完全沒有既有受眾的帳號不友善——排名不是看
原始 upvote 數，是看投票者帳號的權重，需要在發布前就先累積一批會在
發布當天真心投票、留言的人。務實做法：

- **不要在還沒有任何早期使用者/追蹤者時發 Product Hunt**——這是常見的
  順序錯誤：Product Hunt 該是「已經有一小群人认识你、且真心喜歡產品」
  之後拿來擴大聲量的渠道，不是冷啟動渠道。
- 排序建議：先做 Show HN（第 4.1 節）與持續的 Twitter/dev.to 內容
  （第 4.2、4.5 節），等累積到有一定數量的早期使用者/GitHub star/
  waitlist 訂閱者後，再排 Product Hunt，且發布前主動私訊已經表達過
  興趣的人「這週會上 Product Hunt，如果喜歡的話這天麻煩幫忙看看」——
  這不是灰色地帶的操縱票數，是任何 Product Hunt 攻略都會建議的正常
  作法，跟 HN 的 vote brigading（協調匿名新帳號洗票）性質完全不同。

## 5. 內容與素材：demo 就是 pitch

對 onagent 這種「AI 真的操作 UI」的產品，**看得到動作的 demo 比任何
文案都更有說服力**——這正是 Stripe、Twilio 這類 API 產品「文件即行銷」
的邏輯延伸：Stripe 的 quickstart 讓開發者幾分鐘內看到一筆測試交易真的
跑通、產生「啊哈」時刻；onagent 的對應版本應該是**讓人在幾分鐘內看到
自己畫面上的按鈕被 AI 按下**。

### 核心資產優先順序（照製作順序，不是重要性排序——每一項都建立在前一項上）

1. **一支 30 秒錄影，用 `demo-app.yaml` 現成的四個 tool**：文字輸入
   「幫我找戶外用品」→ 畫面上搜尋框自動填入並送出 → 「帶我去該分類」
   → 頁面自動跳轉 → 「這個商品不錯，圈起來給我看」→ 對應商品被框線
   高亮 → 「好，幫我結帳」→ 結帳按鈕自動被按下。這組素材已經不需要
   另外設計情境，YAML 定義都在 repo 裡，唯一要做的是找一個像樣的
   demo 電商頁面（`examples/react-demo` 應該已經有雛形）錄下來。
2. **console Playground 的錄影**：強調「不用寫任何前端程式碼，就能先
   感受一次 AI 怎麼呼叫你定義的 tool」——這是降低試用門檻的關鍵素材，
   對應到 Stripe sandbox「不用先接好完整系統就能看到東西動起來」的
   同一個心理機制。
3. **Claude Code Skill 的錄影**：展示「開發者只需要請 Claude 用
   onagent skill 建立 app、定義 tool、推送到後台」全程幾乎不用手動
   打指令——這是目前市場上同類產品幾乎沒有的機制，值得單獨做一支素材
   而不是埋在通用 demo 裡。
4. **互動式 demo 頁面**（優先度較後，但長期投資報酬率最高）：把 demo
   影片裡的那個電商頁面直接公開部署成一個任何人都能上去打字互動的
   頁面，而不只是看影片——這比影片更有說服力（影片可以被懷疑是剪輯過
   的，互動頁面沒辦法作假），但需要先確保 Cloud Run 後端撐得住陌生
   流量、且有基本的濫用防護（例如速率限制），這部分等第 6 節時間軸
   排到 Show HN 前置準備時再一併處理。

## 6. 前三個月的執行時間軸

假設從現在開始算起，且執行者是一個人、業餘或半業餘投入。時間軸刻意
寬鬆，抓的是「合理不會爆掉」的步調，不是理想化的衝刺排程。

### 第 1 個月：把現有素材整理成看得懂的入口，先在小範圍驗證

- 錄第 5 節列的核心 demo 影片（第 1、2 項），放上 landing page 與
  GitHub README 最上方。
- 開 GitHub Discussions，README 補一個「有問題來這裡問」的連結。
- 開始每週 2-3 則 Twitter 技術/build-in-public 貼文（取材自
  `docs/security-and-transport.md`、`docs/subscription-usage-quota-
  design.md` 裡已經寫好的技術決策）——目的不是衝粉絲數，是先建立
  「這個帳號認真在做技術」的既有紀錄，這樣之後 Show HN/Product Hunt
  發文時，點進帳號主頁的人會看到一致的技術可信度，而不是一個突然
  冒出來的新帳號。
- 私下找 5-10 位認識的、真的做 web 開發的朋友/前同事試用完整流程
  （建 app、定義 tool、串 SDK），蒐集「卡在哪」的真實摩擦點——這比
  任何行銷渠道都重要，因為 Show HN/Product Hunt 只有一次好球，若卡點
  沒修就先大範圍曝光等於浪費那次機會。

### 第 2 個月：擴大內容產出、修復第 1 個月發現的摩擦點、準備 Show HN

- 根據第 1 個月的私下試用回饋，修掉最阻礙上手的問題（常見情境：文件
  某一步跳過了沒講清楚、CLI 錯誤訊息不夠明確、Playground 某個互動
  沒有即時回饋）。
- 發布 1-2 篇 dev.to/部落格文章（第 4.5 節列的素材）。
- 錄製第 5 節第 3 項（Claude Code Skill）demo。
- 開始準備 Show HN 發文：標題草稿多寫幾個版本、確認 demo 連結（互動
  頁優先，其次 repo）在流量尖峰下不會掛掉、抓一個週二到週四美東上午
  的發文時段。
- 若第 1 個月的 Twitter 內容已經有一定迴響（例如穩定有陌生帳號主動
  轉發/回覆），可以開始蒐集「若正式上線願意收到通知」的 waitlist
  email——不需要另外做落地頁,一個簡單的表單或甚至一則置頂推文附
  Google表單就夠,重點是開始累積一份「這些人明確表達過興趣」的名單,
  這份名單會直接影響第 3 個月 Product Hunt 發文當天的初始動能。

### 第 3 個月：Show HN、視狀況接續 Product Hunt

- 執行 Show HN 發文（第 4.1 節的完整檢查清單）。發文當天清空行事曆，
  全程盯留言。
- 發文後 1-2 週內寫一篇「Show HN 上線後學到的事」追蹤文章——不論結果
  好壞都值得寫，這類覆盤文章本身也是很好的 build-in-public 素材，且
  對之後任何想做類似發文的其他開發者也有實際參考價值，容易被轉發。
- 若第 2 個月累積的 waitlist/追蹤者已經有一定規模（沒有絕對數字門檻，
  但至少要感覺「這批人發布當天真的會來捧場」），排 Product Hunt
  發文，發布前主動私訊 waitlist 名單裡的人。
- 這個月**不**該做的事：不要因為 Show HN 反應不如預期就急著加購買
  廣告/找 KOL 推廣衝量——現階段還沒有計費機制,付費導流進來的人轉換
  不到任何東西,只會拉高流量成本卻沒有對應的漏斗接住,詳見第 7 節。

## 7. Launch 時機判斷：現在能不能打大的？

**不能，且不應該現在打大的——但原因不是「產品還沒做完」，是「後面的
漏斗接不住」。**

onagent 現在缺的不是「能不能玩」（console、Playground、SDK 都是真的
能動手串接），而是缺「玩完之後呢」——沒有訂閱方案、沒有用量配額，代表
現在衝進來的每一個使用者，事實上都在用一個沒有商業模式支撐的免費服務。
這不是說現在不能對外曝光，而是曝光的**目的**要調整：

- **現在該追的目標**：讓對的一小群人（ICP 描述的那群開發者）知道這個
  東西存在、覺得有意思、留下聯絡方式或動手玩一次、記住這個名字。
- **現在不該追的目標**：轉換率、付費使用者數、大量陌生流量——沒有計費
  機制時衝大量流量,除了拉高 Cloud Run 帳單、增加濫用風險（見
  `docs/security-and-transport.md` 裡尚未做的 per-session 授權)之外,
  不會換到任何長期資產。

這跟 Supabase 的 Launch Week 邏輯剛好相反但同樣務實：Supabase 是在
「已經有底層產品」的前提下，每季固定用一週密集曝光新功能；onagent
現在更接近 Supabase **更早期**的階段——先靠早期使用者一對一回饋把
產品打磨到位、建立起技術可信度，"launch week" 式的大規模週期性曝光
應該留到訂閱/配額系統上線、真的有東西可以「轉換」之後再做。

**具體判斷準則**：等 `docs/subscription-usage-quota-design.md` 裡的
配額系統(哪怕只是最小可行版本——一個簡單的月配額上限、超過就回傳
`quota_exceeded` 錯誤碼)真的上線,才是適合認真考慮"大 launch"（大量
付費導流、KOL 合作、正式新聞稿等級的曝光）的時間點。在那之前,所有
渠道操作都該定位成「建立心智佔有率與早期社群」,不是「拉轉換」。

## 8. 指標與檢查點

現階段不追營收指標（還沒有計費），追的是能預示「產品是否值得繼續投入
行銷心力」的先行指標：

- **GitHub**：star 數成長曲線（尤其是 Show HN 發文前後的斜率變化）、
  Discussions 裡的提問品質（有人願意花時間問細節,代表真的在認真評估,
  比純粹的 star 數更有意義)。
- **Twitter**：不是 follower 數本身,是「有陌生帳號主動轉發/回覆技術
  內容」的頻率——這代表內容真的打中了圈外的人,不只是既有人脈互相
  捧場。
- **waitlist/早期使用者名單**：累積速度、以及（若能追蹤）有多少人
  真的動手完成過一次「建 app → 定義 tool → 串 SDK」的完整流程,而不
  只是留了 email 就沒下文——完成過完整流程的人,才是配額系統上線後
  最可能第一批轉換的對象。
- **Show HN / Product Hunt 單日流量**：對應到第 6 節的兩個一次性事件,
  事後回顧的重點不是當天流量峰值,而是峰值過後 1-2 週的「長尾」——是否
  持續有人透過搜尋或分享找過來,這比單日高峰更能反映內容/demo 本身的
  持久吸引力。

## 9. 資源限制下的取捨原則

整份文件反覆出現的一個判斷邏輯,在這裡明講出來,方便之後任何新增的行銷
決定都能套用同一套標準：

1. **能重複利用既有素材的優先做**——`docs/` 底下的技術決策文件、
   `demo-app.yaml` 現成的 demo 情境、README 已經寫好的架構說明,這些
   都是零邊際成本的行銷素材,應該優先被改寫/錄製,而不是另外發想新
   內容。
2. **一次性、高槓桿的事件（Show HN、Product Hunt）只在準備好時打**，
   且一旦打了要全程盯著——沒有全職行銷人力時,分散精力做五個渠道的
   半殘操作,不如集中做兩三個渠道的完整操作。
3. **不開維護不了的渠道**——Discord 是最容易被無條件建議、但對現在
   規模最容易變成負資產的例子（第 4.4 節)。任何新渠道開通前先問
   「我真的有時間持續維護這個嗎」,答案不確定時,先不開,用現有渠道
   （GitHub Discussions）撐著就好。
4. **付費獲客排在配額系統上線之後**——現在没有東西可以「轉換」,付費
   流量只是拉高成本、不會留下資產,所有現階段的努力都該以「免費、
   靠內容/demo/口碑」為主。
5. **誠實比完美更有轉換力**——README 的「還沒做什麼」章節、
   `docs/security-and-transport.md` 主動承認的安全缺口、這份文件本身
   對「現在能不能打大的」給出的保留答案,都是刻意的選擇。開發者受眾
   對過度包裝高度敏感,對誠實列出限制的專案反而更願意給出真實回饋、
   也更可能成為早期真心推薦的人。
